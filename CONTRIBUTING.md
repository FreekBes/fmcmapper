# Contributing to fmcmapper

Thanks for helping out! By far the most common contribution is **adding support
for a new Minecraft version**, so this guide focuses on exactly that. (For bug
fixes and features, the same setup applies — jump to [Testing](#testing-your-changes)
and [Pull requests](#opening-a-pull-request).)

## Development requirements

- **Git** and a **GitHub account** — to clone the repo and open pull requests.
- **Docker** + **Docker Compose** — the easiest path. The `docker-compose.dev.yml`
  and `docker-compose.dump.yml` stacks build everything in containers, so you
  don't need Node or a JDK installed locally. <https://docs.docker.com/get-docker/>

Only needed if you build a component **outside** Docker:

- **Node.js 24+** and **npm** — to build and run the renderer (`src/`) directly.
- **A full JDK 25** (not just a JRE — the build needs `javac`) — to build the
  map-color-dump mod (`MapColorDumpMod/`). The Gradle wrapper is bundled, so you
  don't need to install Gradle separately.

## How version support works

fmcmapper is tied to a specific Minecraft version in two ways, and both have to
be updated together:

1. **Colour tables** — `assets/map_colors.json` and `assets/biome_colors.json`
   hold every block's map colour and every biome's grass/foliage/water tint.
   They're generated by running the companion **map-color-dump** mod on a
   **Minecraft server** (in [`MapColorDumpMod/`](MapColorDumpMod/)) and differ
   between versions as Mojang adds/changes blocks and biomes.
2. **The renderer's target version** — `src/buildtiles.ts` records which version
   the bundled tables were built for, so it can warn on a mismatch.

The renderer (TypeScript) is mostly version-agnostic. The **mod** is not: it's
written against one Minecraft version's API, so a new version often needs small
code changes (see [step 2](#2-make-the-mod-compile-against-the-new-version)).

## Adding support for a new Minecraft version

Work in this order — each step depends on the previous one.

### 1. Update the MapColorDumpMod's build versions

Edit [`MapColorDumpMod/gradle.properties`](MapColorDumpMod/gradle.properties).
Set `minecraft_version`, then get the matching Fabric versions from
<https://fabricmc.net/develop>: pick your Minecraft version there and it lists
the recommended **loader**, **Fabric API**, and **loom** versions (it even shows
a ready-made `gradle.properties` snippet). Copy those into:

- **`loader_version`**
- **`fabric_api_version`** (the one ending in `+<mcversion>`, e.g. `0.153.0+26.2`)
- **`loom_version`**

### 2. Make the mod compile against the new version

Mojang renames classes/methods between versions, so
[`MapColorDump.java`](MapColorDumpMod/src/main/java/es/freekb/mapcolordump/MapColorDump.java)
may not compile as-is. Build the mod image and fix what breaks:

```
docker compose -f docker-compose.dump.yml build
```

> Optionally, you can skip Docker and build the mod directly (needs a full
> JDK 25): `cd MapColorDumpMod && ./gradlew build`

Known churn to expect:

- registry lookup helpers (e.g. `registryOrThrow` → `lookupOrThrow`),
- the `MapColor` class and its fields,
- biome colour accessors (`getGrassColor`, `getFoliageColor`, …).

Since Minecraft 26.1 the game ships **unobfuscated** — the jar already carries
the real class/method names, so there are no mappings and loom doesn't remap
anything (that's why `build.gradle` has no `mappings` line, and uses
`implementation`/`jar` rather than `modImplementation`/`remapJar`). Breakages
are ordinary Mojang renames/refactors, not mapping changes — just update the
code to the new names.

### 3. Regenerate the colour tables

With the mod building, run it to dump fresh tables straight into `assets/`:

```
docker compose -f docker-compose.dump.yml up --build
```

This starts a headless dev server, fetches the colormap textures, and writes
`assets/map_colors.json` + `assets/biome_colors.json`. Commit both files.
(Details and the non-Docker route are in
[`MapColorDumpMod/README.md`](MapColorDumpMod/README.md).)

### 4. Update the renderer's target version

In [`src/buildtiles.ts`](src/buildtiles.ts):

```ts
const TARGET_VERSION = '26.2';      // -> the new Minecraft version
const TARGET_DATA_VERSION = 4903;   // -> the new Minecraft version's DataVersion
```

`TARGET_DATA_VERSION` is the world format's `DataVersion`. You can read it from
`Data/DataVersion` in a `level.dat` created by the new version (the renderer
already parses this), or look it up on the [Minecraft Wiki's data-version list](https://minecraft.fandom.com/wiki/Data_version).
This value only drives the mismatch warning, but keep it correct.

If the new version introduces blocks that should be biome-tinted (new leaves,
plants, etc.) or changes which blocks tint, also update the `TINTS` table in
[`src/chunkmap.ts`](src/chunkmap.ts).

The renderer fingerprints its colour inputs into the manifest (the **render
signature**) and does a full redraw whenever that fingerprint changes — so
existing maps are redrawn instead of keeping stale tiles. It detects these
**automatically**: the colour tables (`assets/*.json`), the resolved `MAP_*`
settings (env value *or* default, from [`src/renderconfig.ts`](src/renderconfig.ts)),
and the `TINTS` table. So editing tints or a brightness default needs **no**
version bump.

The one thing the signature can't see is the coloring **algorithm** itself —
the shading math, the blur, the water-depth formula, the fallback colour. If you
change that logic, **bump `RENDER_VERSION` in [`src/buildtiles.ts`](src/buildtiles.ts)**.
(And leave `MANIFEST_VERSION` alone unless the on-disk tile/biome/manifest layout
changes.)

### 5. Test it

See [Testing](#testing-your-changes) below. At minimum, render a world from the
new version and confirm there's **no version-mismatch warning** and the colours
look right.

## Testing your changes

Use the dev stack — it builds the renderer from your checkout **and includes a
Minecraft server container** ([itzg/minecraft-server](https://github.com/itzg/docker-minecraft-server)),
so you get a live world to render without supplying your own:

```
docker compose -f docker-compose.dev.yml up --build
```

Set the server's `VERSION` in [`docker-compose.dev.yml`](docker-compose.dev.yml)
to the version you're adding so it generates a world of that version. Then open
<http://localhost:8080>.

> Optionally, you can skip Docker and run the renderer directly with Node 24+:
> ```
> npm install
> npm run build
> WORLD_PATH=/path/to/world OUTPUT_PATH=./output npm start
> ```

Check that:

- startup prints **no** "world reports Minecraft … but this renderer was built
  for …" warning,
- grass, foliage, leaves, and water are tinted correctly,
- any blocks new to the Minecraft version render with sensible colours,
  preferably matching the in-game map item (which should already be the case if
  you used the MapColorDumpMod to generate the color tables).

## Opening a pull request

Opening a PR fills in our [template](.github/pull_request_template.md)
automatically — it has a section with the version-bump checklist below (delete
it if your PR isn't a version bump).

1. Fork the repository, and optionally branch off `master` to make your changes.
2. Mention the target Minecraft version in the PR title/description, and note
   any mod code changes you had to make for it (helps the next version bump).
3. Commit message *wording* doesn't matter for CI — image versions come from a
   build number, not commit conventions. What matters is **which files change**:
   - changes under `MapColorDumpMod/**` rebuild the `mapcolor-dump` image,
   - changes to renderer files (`src/**`, `assets/**`, `Dockerfile`, …) rebuild
     the `fmcmapper` image.
4. On merge, the workflows automatically build and push images tagged with the
   new Minecraft version (read from `TARGET_VERSION` / `minecraft_version`), so
   you don't need to touch `.github/workflows/`.

**Using AI is welcome.** You're free to use AI tools to help write the code or
the pull request — as long as the result is clear and you stand behind it: the
code should be understandable, the added functionality clearly explained, and
the PR description should make sense on its own. Review anything an AI generates
before submitting; you're responsible for what you open.

## Live demo (maintainers)

The [live demo](https://freekbes.github.io/fmcmapper/) is published by the
[`deploy-pages.yml`](.github/workflows/deploy-pages.yml) workflow, which
downloads an example world, renders it, and deploys to GitHub Pages — so no
world or tiles are committed to the repo.

The live demo always uses the latest `master` build and a preconfigured world.

## Where things live

| Path | What it is |
|------|------------|
| `src/buildtiles.ts` | Renderer entry point; `TARGET_VERSION` / `TARGET_DATA_VERSION`, plus `RENDER_VERSION` / `MANIFEST_VERSION`, live here. |
| `src/chunkmap.ts` | Block/biome → colour logic, including the `TINTS` table. |
| `src/biomevector.ts` | Generates the biome polygons for a region, used for the tooltip in the map viewer. |
| `src/worker.ts` | Renders one region (worker thread). |
| `src/viewer.ts` | Generates the Leaflet map viewer (`index.html`). |
| `src/renderconfig.ts` | Resolves the `MAP_*` env vars and defaults. |
| `assets/*.json` | Generated colour tables (commit regenerated ones). |
| `MapColorDumpMod/` | The Fabric mod that produces the colour tables. |
| `.github/workflows/` | CI that builds and pushes the images. |
